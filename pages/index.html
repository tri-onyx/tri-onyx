Core Features

Every layer designed to limit what a compromised agent can do with what it knows.

🔒

Isolated Agent Containers

Each agent runs in its own Docker container with a per-agent FUSE filesystem, network rules, and no shared state. Compromise one agent, the rest don't know.

🧬

Taint & Sensitivity Tracking

Two independent axes — Biba integrity for trustworthiness, Bell-LaPadula confidentiality for sensitivity — track what each agent has been exposed to.

🛡️

Information Flow Enforcement

The gateway intercepts all inter-agent communication and blocks flows that would violate integrity or confidentiality constraints. No bypass possible.

📂

FUSE Filesystem Driver

A Go passthrough filesystem enforcing per-file read/write policies. Pre-computed path trie for O(1) access decisions. Symlinks unconditionally denied.

🔍

Auditable Everything

Structured logs for every file access, tool call, message route, and policy violation. Complete provenance tracking with risk manifests.

🌐

Graph Analysis

Transitive risk propagation across the agent topology. Detects multi-hop policy violations that pairwise checks would miss.

Design Philosophy

Principles that guided every architectural decision.

⚠️

Information is the threat

An LLM that has ingested a prompt injection is dangerous regardless of what tools it has. Track what enters context, and where it flows next.

🚫

No LLM in the security boundary

The gateway is entirely deterministic — no AI, no autonomy, no judgment calls. Security decisions are made by code, not models.

📉

Risk reduction, not elimination

The security model makes attacks harder and detectable, not theoretically impossible. Honest about what's achievable.

🔏

Gateway holds all credentials

Agents never touch API keys or secrets directly. The gateway injects credentials at call time, preventing exfiltration.

⬆️

Monotonic exposure

Taint and sensitivity only increase during a session. Once an agent has seen something, it can't un-see it. Simple, predictable, auditable.

👤

Single operator

Built for one person running their own agents. No multi-tenancy complexity. Aligns trust model with reality.

Quick Start

Up and running in minutes. Just Docker.

terminal 
# Clone the repository
$ git clone https://github.com/tri-onyx/tri-onyx.git
$ cd tri-onyx

# Build the gateway image
$ docker build -f gateway.Dockerfile -t tri-onyx-gateway:latest .

# Build the agent runtime image
$ docker build -f agent.Dockerfile -t tri-onyx-agent:latest .

# Start the gateway
$ docker run --rm -p 4000:4000 \
    -v $(pwd):/app -w /app \
    -v /var/run/docker.sock:/var/run/docker.sock \
    -e TRI_ONYX_HOST_ROOT=$(pwd) \
    --env-file .env \
    tri-onyx-gateway:latest mix run --no-halt

Frequently Asked Questions

Docker provides process isolation, but it doesn't track what information flows between containers. TriOnyx adds a security layer on top of Docker: per-file FUSE policies, information flow tracking across agents, taint propagation detection, and a gateway that intercepts all inter-agent communication to enforce integrity and confidentiality constraints.
TriOnyx uses the Claude Agent SDK for the agent runtime. Agents are driven by Claude models (Sonnet, Opus, Haiku). The gateway itself is non-agentic — no LLM is used in the security boundary.

If a high-taint agent tries to send data to a clean agent, the Bandwidth-Constrained Trust Protocol kicks in, limiting what can pass.
The FUSE driver uses a pre-computed path trie for O(1) access decisions, so the per-call overhead is minimal. Policy expansion happens once at mount time. For typical agent workloads (reading source files, writing results), the overhead is negligible compared to LLM API latency.
TriOnyx is designed for single-operator use — one person running their own agents. It's built on Elixir/OTP (known for reliability) and the security model is well-documented with formal Architecture Decision Records. It's actively developed and used by its creators. Evaluate it against your own requirements.
Agents are markdown files with YAML frontmatter in workspace/agent-definitions/. The frontmatter declares permissions (tools, filesystem access, network policy) and the body is the system prompt. The gateway translates these into Docker container configuration, FUSE policy, and iptables rules.
